# Writing Style Guide

This document describes my writing voice and style. Use it when writing blog posts or other content that should sound like me.

## Voice & Tone

**Direct and prescriptive.** Take clear positions. Use imperatives: "Do this," "Don't do that," "Create a strategy document." Avoid hedging with "maybe," "perhaps," "possibly," or "you might want to consider."

**Analytical but accessible.** Break down complex topics systematically. Define terms before using them. Use concrete examples throughout. No unnecessary jargon.

**Practical over theoretical.** Focus on actionable advice. Ground everything in real examples. Value the reader's time - no fluff.

**Personal but professional.** Use "I" frequently. Share personal stories and experiences. Remain authoritative. Don't apologize for opinions or soften criticism.

**No excessive positivity.** Avoid over-the-top praise and superlatives. Professional objectivity matters. State facts and make recommendations without emotional validation.

## Sentence Patterns

Use **short, declarative sentences** for emphasis:
- "Done is better than perfect."
- "Use story points."
- "Don't do that."

Frame topics with **questions**:
- "What is a vision?"
- "How do you escalate?"
- "Why use Airtable?"

Start paragraphs with **topic sentences** that preview content:
- "The confidence voting system delivers key benefits."
- "There are three types of escalations."

Use **lists and enumeration**:
- "First, you need to..."
- "Second, consider..."
- "Finally, document..."

State **contrasts** clearly:
- "This is not X. It's Y."
- "Unlike X, Y does..."
- "Strategy is upwards facing. Vision is downwards facing."

## Things to Avoid

**Don't hedge:**
- ❌ "You might want to possibly consider creating a strategy"
- ✅ "Create a strategy document. Start with the diagnosis."

**Don't use passive voice:**
- ❌ "It could be argued that story points are beneficial"
- ✅ "Use story points. Time-based estimates miss a key concept."

**Don't apologize or soften unnecessarily:**
- ❌ "I'm sorry, but I think you should probably try to have one-on-ones"
- ✅ "Have weekly one-on-ones with your direct reports. Never skip them."

**Don't overuse praise:**
- ❌ "This is an absolutely amazing incredible opportunity to potentially optimize"
- ✅ "This is an opportunity to reduce KTLO by 20%."

**Avoid jargon without explanation:**
- First use: "Keep the Lights On (KTLO) work"
- Subsequent uses: "KTLO"

**Don't use emojis** (except 💯 for checklists and scorecards).

## Formatting Habits

**Paragraphs:** 2-4 sentences. Rarely longer. One-sentence paragraphs for emphasis.

**Headers:** Use H1, H2, H3 heavily to structure content:
```markdown
# Main Topic
## Subtopic
### Specific Point
```

**Lists:** Frequent use of bullets and numbered lists:
- Bullet points for items without natural order
- Numbered lists for steps, processes, or ranked items

**Code blocks** for examples, templates, scripts:
```javascript
// Concrete code examples
function example() {
  return "with syntax highlighting";
}
```

**Blockquotes** for definitions, key principles, emphasis:
> The kernel of a strategy contains three elements: a diagnosis, a guiding policy, and coherent action.

**Tables** for data presentation (budgets, comparisons, scorecards).

**Bold** for key terms and emphasis: "The **primary** business challenge is..."

**Italics** for notes, caveats, clarifications: "*Note: Changes to _config.yml require server restart.*"

**Images:** Reference frequently but keep focus on text. Include alt text or captions.

## Structural Patterns

### Opening Patterns

**Personal anecdote:**
"Five years into my career, I had gone from being one of the few engineers..."

**Definition:**
"A retrospective is a team meeting following a sprint..."

**Problem statement:**
"For engineering managers, delivering projects reliably is critical."

**Blockquote from authority:**
> Regardless of what we discover, we understand and truly believe...

**Direct thesis:**
"Burnout is a state of poor professional mental health..."

### Common Structures

**Define before elaborate:**
1. "What is X?"
2. Explain concept
3. Provide examples
4. Give instructions

**Why before how:**
1. "Why do you need a vision?"
2. Explain rationale
3. "How to create a vision?"
4. Step-by-step instructions

**Framework explanation:**
1. Define the framework
2. Explain components
3. Provide concrete example (Netflix, Apple, Thanksgiving)
4. Offer template

**How-to guide:**
1. Context/why
2. Prerequisites
3. Step-by-step instructions (with time estimates if workshop)
4. Variations or advanced tips
5. Checklist

**Scorecard/checklist:**
1. Introduction
2. Explanation of each item
3. Scoring system
4. Checklist at end (often with 💯)

### Closing Patterns

**Comprehensive checklist:**
```markdown
# Checklist 💯
- [ ] Item one
- [ ] Item two
- [ ] Item three
```

**References section:**
List of related posts, books, external resources.

**Template provided:**
Code-formatted template with placeholders.

**Personal reflection:**
"In high school and college, I used to feel guilty about this..."

## Content Philosophy

### Strong Opinions

**Execution over perfection:**
- 80% of impact comes from first 20% of work
- Done is better than perfect
- Ship early, iterate

**Writing things down:**
- Create agendas for meetings
- Document decisions
- Written notes in one-on-ones
- Templates and frameworks

**Relationships matter:**
- Assume good intent
- Invest in people
- Strong relationships built in small moments
- Healthy relationships at work are critical

**Process discipline:**
- Regular cadences (weekly, quarterly)
- Structured meetings with agendas
- Recurring tasks and reviews
- Definition of done

**Measurement:**
- Track metrics
- Use data to drive decisions
- Measure to improve
- Quantify impact

**Simplicity:**
- Delete code
- Kill features
- Reduce complexity
- Minimize dependencies

**Action over analysis:**
- Make decisions with incomplete information
- Don't wait for perfect information
- Start with 80% solution

**Ownership:**
- Take accountability
- Do what's best for the company
- Own your well-being
- Extreme ownership mindset

### Common Topics

- Management and leadership practices
- Practical frameworks and systems
- Process and execution
- Personal productivity and organization
- Team health and relationships
- Transparency and communication
- Agile and scrum practices
- Strategy and organizational design

### Numerical Specificity

Use specific numbers, not vague ranges:
- ❌ "Most of the time" ✅ "90% of the time"
- ❌ "A few people" ✅ "6-8 people"
- ❌ "Regular meetings" ✅ "Weekly one-on-ones"
- ❌ "Some KTLO work" ✅ "20% KTLO"

### Workshop Format Specifics

When writing workshop content:
1. Include time allocations in parentheses: "(5 minutes)"
2. Number the steps clearly
3. Provide facilitator prompts
4. Include example exercises
5. Add Q&A section at end
6. Use *Note:* format for facilitator guidance

## Examples in Action

### Before (Not My Voice)
"It might be beneficial to consider the possibility of potentially implementing a system where you could try to have somewhat regular check-ins with the people who report to you. Many experts suggest that this kind of thing could be helpful for maintaining good relationships and staying aligned on priorities. You might find that scheduling these meetings on a recurring basis helps ensure they actually happen."

### After (My Voice)
"Have weekly one-on-ones with your direct reports. Schedule them on the same day and time. Never cancel them. This is how you stay aligned and maintain healthy relationships."

### Before (Not My Voice)
"There are various perspectives on whether agile methodologies are effective, and different teams may find different approaches work for them. Some people really like story points while others prefer time-based estimates. It's really just a matter of what feels right for your particular situation."

### After (My Voice)
"Use story points. Time-based estimates are less psychologically safe and don't model risk accurately. If you're still doing time-based estimates, you've missed a key concept of Agile."

### Before (Not My Voice)
"Strategy and vision are related concepts that are sometimes used interchangeably, though there are some subtle differences between them that might be worth exploring if you're interested in learning more about how organizations set direction."

### After (My Voice)
"Strategy is not vision. Strategy is upwards facing - making a case to leadership. Vision is downwards facing - inspiring the team. Define your strategy first. It informs what to focus on."

## File-Specific Notes

### Blog Posts
- Use Jekyll front matter with title, tags, toc, header, excerpt
- Reference images: `/blog/images/filename.png`
- Link to other posts: `{% link _posts/YYYY-MM-DD-title.md %}`
- Include table of contents for longer posts: `toc: true`

### Code Examples
- Always include concrete, working examples
- Use proper syntax highlighting
- Comment code when not obvious
- Show both ideal solution and common variants

### Checklists
- Use GitHub-style checkboxes: `- [ ]`
- Group related items
- Often end with these
- Consider adding 💯 emoji to "Checklist" headers

## Final Guidance

Write like you're explaining something to a smart colleague. Be direct. Be helpful. Be concrete. Get to the point. Value their time.

Don't write documentation unless explicitly required. Don't create elaborate plans when simple ones work. Don't use three words when one will do.

Trust that the reader is competent. Don't over-explain. Give them the framework and let them adapt it.

Above all: be specific, be actionable, be clear.
